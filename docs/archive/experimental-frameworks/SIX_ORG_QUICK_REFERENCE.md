### 每日会议

| 会议 | 时间 | 参与者 | 时长 | 目标 |
|------|------|--------|------|------|
| **Daily Standup** | 09:00 | 实施组+PMO+QA代表 | 15分钟 | 进度同步、阻碍识别 |

## 🎯 决策流程速查

### 决策层级

```
战略决策层 (顾问组 + 架构师组)
  → Go/No-Go、方向调整、重大变更

战术决策层 (PMO + 架构师组)
  → 资源分配、优先级、里程碑

执行决策层 (实施组 + QA组)
  → 技术方案、实现方式、Bug处理
```

### Go/No-Go 决策流程

```
1. 评估准备 (实施组 + 架构师)
   └→ 技术可行性、风险评估、成本分析

2. 质量把关 (QA组)
   └→ 测试覆盖、质量指标、遗留问题

3. 进度评估 (PMO)
   └→ 进度偏差、资源可用、依赖检查

4. 顾问评审 (顾问组) - 可选
   └→ 战略一致、长期影响、外部风险

5. 决策会议 (所有组织)
   └→ 立场陈述、讨论辩论、最终决策

6. 决策记录和沟通
   └→ 决策日志、理由记录、全员沟通
```

### 升级路径

```
Level 1: 组织内决策
   ↓ (遇阻)
Level 2: 跨组织协调 (PMO/架构师)
   ↓ (无法解决)
Level 3: 战略决策 (顾问组 + PMO)
   └→ 48小时内响应
```

## 🔧 工具速查

### 项目管理

**GitHub Projects:**
- Kanban Board: Backlog → To Do → In Progress → Review → Done
- 标签: P0/P1/P2/P3, type, org:xxx
- Milestone: Week X, Month Y

**Issue 模板:**
- Feature (新功能)
- Bug (Bug修复)
- Tech Debt (技术债务)

### 代码协作

**Code Review 流程:**
1. 创建 Pull Request
2. 自动检查通过 (CI/CD)
3. 至少 1 个 Reviewer 批准
4. 修改请求已解决
5. 合并到主分支

**CI/CD Pipeline:**
- Type Check → Lint → Unit Test → Coverage → Build
- 预计时间: <5分钟
- 通过率目标: >95%

### 沟通工具

**Slack 频道:**
- #general - 全团队公告
- #daily-standup - 每日站会
- #code-review - CR 通知
- #ci-cd - CI/CD 通知
- #org-xxx - 各组织私密频道

**自动通知:**
- PR Opened → #code-review
- PR Merged → #dev-activity
- CI Failed → #ci-cd + @mention
- Issue Created → 根据标签路由

## 🎯 快速决策矩阵

### 谁有权决策？

| 决策类型 | 决策者 | 是否需要批准 | 是否需要记录 |
|---------|--------|-------------|-------------|
| **日常技术方案** | 实施组 | 否 | 否 |
| **技术选型** | 架构师组 | 是 (PMO) | 是 |
| **优先级调整** | PMO | 否 | 是 |
| **架构变更** | 架构师组 | 是 (顾问组) | 是 |
| **Go/No-Go 发布** | PMO + QA组 | 是 (顾问组) | 是 |
| **流程变更** | 复盘组 | 是 (PMO) | 是 |

### 决策记录阈值

**必须记录的决策:**
- 影响范围 > 1 个模块
- 成本影响 > ¥10,000
- 时间影响 > 1 周
- 涉及多个组织

**记录位置:** `docs/decisions/YYYY/DEC-YYYY-NNN.md`

## 🚨 紧急情况处理

### 定义

**P0 紧急情况:**
- 线上故障
- 数据丢失
- 安全漏洞
- 严重性能问题

### 响应流程

```
1. 发现 (任何人)
   ↓ @mention PMO + 技术负责人
2. 确认 (PMO + 架构师)
   ↓ 评估严重程度
3. 组织响应团队
   ↓ 实施组 + QA组 + 架构师
4. 执行修复
   ↓ 实施组主导
5. 验证
   ↓ QA组验证
6. 复盘
   └→ 复盘组主持 (24小时内)
```

### 沟通渠道

- **紧急频道:** #emergency
- **响应时间:** <15分钟
- **更新频率:** 每30分钟

## 🎓 最佳实践速查

### 会议效率

- ✅ 提前24小时发布议程
- ✅ 会议时长严格控制在计划内
- ✅ 专人主持，专人记录
- ✅ 明确行动项和责任人
- ❌ 避免无议程的会议
- ❌ 避免超过1小时的会议（除特殊情况）

### Code Review

- ✅ 至少 1 个 Reviewer 批准
- ✅ 所有 CI 检查通过
- ✅ 响应时间 <24小时
- ✅ 建设性反馈意见
- ❌ 避免过度纠结风格问题

### 文档维护

- ✅ 代码变更同步更新文档
- ✅ 重要决策记录在案
- ✅ 文档保持新鲜 (<7天)
- ❌ 避免文档过时
- ❌ 避免文档冗余
